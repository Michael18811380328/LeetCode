# JavaScript LeetCode 仓库重构方案

这个仓库已经能正常承载大量题解和测试，但目前更像“长期堆叠出来的题解集合”，而不是一个结构清晰、规则统一的项目。下面按优先级整理重构建议，目标是先减少维护成本，再提升一致性和可扩展性。

## P0 先解决配置冲突和基础稳定性

1. 统一 ESLint 配置入口
   - 现在同时存在 `eslint.config.js` 和 `.eslintrc.json`
   - 这会让规则来源不清晰，容易出现“本地能过、CI 不能过”或相反的情况
   - 建议只保留一种配置体系，并把规则集中到一个地方

2. 明确测试运行方式
   - `package.json` 里既有 Jest，又有 `standard`，还有 Babel、Rollup、Parcel
   - 工具链过多会让新题目维护成本变高
   - 建议确定主测试入口只保留一套，例如：`jest + eslint`

3. 检查 `src/leetcode.js` 的导出完整性
   - 这个文件目前只手动导出了一小部分题解
   - 如果它是项目对外入口，就必须保持完整、同步
   - 如果不是必要入口，建议弱化它的存在，避免变成“半维护状态”的聚合文件

4. 修正测试和实现的模块风格差异
   - `test/tree-node.js` 使用 `module.exports`
   - 其他测试文件使用 `import`
   - 这类 CommonJS 和 ESM 混用虽然可能能跑，但长期看不利于统一

## P1 整理目录结构

1. 保持题解目录和测试目录的一一对应
   - 当前 `src/` 和 `test/` 已经是按题号配对，这是优势
   - 但文件命名风格不完全统一，后续应该强制规范化

2. 清理不必要的历史文件和重复实现
   - 仓库里有一些文件命名带 `bad`、`ii`、`1/2/3` 等后缀
   - 这些文件如果是历史版本，建议放入明确的实验目录
   - 如果是最终版本，建议保留唯一实现，避免维护时混乱

3. 统一命名规则
   - 建议统一成一种风格：
     - `0001-two-sum.js`
     - `0100-same-tree.js`
     - `0224-calculate.js`
   - 避免混用 `camelCase`、`PascalCase`、连字符、大小写不一致等形式

## P2 提升测试质量

1. 把单测改成参数化风格
   - 现在很多测试文件是手写多个 `expect`
   - 参数化后更容易维护，也更容易补边界值
   - 尤其适合题解仓库里高度重复的输入输出验证

2. 补充边界和异常场景
   - 当前很多测试更偏向样例验证
   - 应至少覆盖：
     - 空输入
     - 单元素输入
     - 重复值
     - 最小/最大边界
     - 无解场景

3. 让测试与题解风格一致
   - 例如树节点测试应统一使用 `TreeNode` 构造器，不要同时混用字面量对象和构造器对象
   - 这样可以避免“测试通过但数据结构行为并不一致”的情况

4. 改善缺失测试检查脚本
   - `scripts/check-unit-test.js` 目前是按文件名粗略匹配
   - 这个思路可用，但建议进一步支持：
     - 排除工具文件
     - 排除实验文件
     - 输出更明确的缺失清单

## P3 统一代码规范

1. 统一导出与导入模式
   - 如果项目继续使用 `import/export`，就尽量不要在局部文件里回退到 `module.exports`
   - 统一后，工具链和编辑器支持会更好

2. 精简脚本命名
   - `test-debug`、`test-specific`、`test-watch`、`check-unit-test`、`standard-fix` 等脚本都能用，但容易分散
   - 建议保留少量高频脚本，例如：
     - `test`
     - `test:watch`
     - `lint`
     - `lint:fix`
     - `check`

3. 降低规则“过度放松”
   - `.eslintrc.json` 里关闭了很多规则
   - 这是为了兼容题解写法可以理解，但也会让一些明显问题难以被发现
   - 建议逐步恢复一部分真正有价值的规则，比如：
     - `no-undef`
     - `no-redeclare`
     - `no-unused-vars`
     - `eqeqeq`

4. 统一注释目的
   - 当前注释里有题目链接、调试说明、规则解释、中文备注等多种风格
   - 建议保留两类即可：
     - 题目来源或链接
     - 对实现有帮助的解释
   - 其余说明尽量放到 README 或重构文档里

## P4 提升仓库可维护性

1. 建立公共辅助模块
   - 例如：
     - `TreeNode`
     - `ListNode`
     - 数组和树的构造辅助函数
     - 常见断言工具
   - 这样测试文件不需要重复定义数据结构

2. 把题解入口标准化
   - 每道题最好只有一个对外主实现
   - 如果保留多个实现，建议明确命名：
     - `twoSumBruteForce`
     - `twoSumHashMap`
     - `twoSumTwoPointers`
   - 不要让 `twoSum1 / twoSum2 / twoSum3` 长期散落而没有说明

3. 统一题解模板
   - 每个新题建议从同一个模板创建
   - 模板中固定包含：
     - 问题编号和名称
     - 导入语句
     - 主实现
     - 复杂度说明
     - 对应测试文件

## P5 工具链收敛

1. 明确主工具组合
   - 当前同时有 Babel、Rollup、Parcel、Jest、ESLint、Standard
   - 对于题解仓库来说，这些工具已经偏多
   - 建议保留最小集合：
     - Jest 用于测试
     - ESLint 用于静态检查
     - Babel 仅在确实需要时保留

2. 文档收敛
   - 可以把零散说明合并成一个简洁的 `README.md`
   - 保留内容建议包括：
     - 如何运行测试
     - 如何新增题解
     - 目录约定
     - 代码规范

3. 自动化检查入口
   - 最好有一个命令可以一次性完成：
     - lint
     - unit test
     - 缺失测试检查
   - 这样可以降低每次提交前的认知负担

## 推荐执行顺序

1. 先统一 ESLint / Jest / CommonJS-ESM 风格
2. 再整理目录和命名
3. 然后补强测试和辅助模块
4. 最后收敛工具链和文档

## 如果只做最有价值的 3 件事

1. 统一 ESLint 配置，去掉重复和冲突规则
2. 把测试改成参数化，并补边界用例
3. 建立公共数据结构模块，减少测试重复代码
