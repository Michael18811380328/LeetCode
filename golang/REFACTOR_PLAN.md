# Golang LeetCode 仓库重构方案

这个仓库的规模不大，但已经能看出典型的刷题项目特征：文件按题号分散、实现和测试并列、包名统一但不够工程化、少量代码风格不一致。下面按优先级整理重构建议，目标是先让仓库更稳定，再让结构更清晰，最后再提升维护体验。

## P0 先修复基础一致性问题

1. 统一包命名策略
   - 目前所有源码和测试都使用 `package utils`
   - 这能跑，但语义不够明确，也不利于后续扩展
   - 建议明确这个仓库到底是“单包题解仓库”还是“按模块分包仓库”
   - 如果继续保持单包，至少要统一命名约定并保持测试和源码一致

2. 统一代码格式
   - 当前部分文件存在明显的排版不一致，例如：
     - `0231-isPowerOfTwo.go` 缩进不统一
     - `0881-num-rescue-boats.go` 有多余分号和空格风格问题
     - `0663-judgeSquareSum.go` 中运算符周围空格不统一
   - Go 项目里格式问题最好直接交给 `gofmt`

3. 明确测试覆盖范围
   - 目前仓库文件数量不多，测试文件和实现文件是一一配对的，这是优点
   - 但测试数量仍然偏少，边界值覆盖有限
   - 建议至少保证每题覆盖：
     - 标准样例
     - 边界输入
     - 最小输入
     - 特殊输入

4. 避免不必要的副作用
   - `1929-getConcatenation.go` 现在实现为 `append(nums, nums...)`
   - 这会复用底层数组，可能修改原始切片
   - 如果函数语义要求“返回新结果且不影响原输入”，应该额外说明或改成更安全的写法

## P1 整理目录和文件职责

1. 明确源码和测试的职责
   - 当前目录下所有题解和测试都放在根目录
   - 对于小仓库可以接受，但随着题目增多会变乱
   - 建议至少拆成：
     - `src/`：题解源码
     - `test/`：单元测试
     - `pkg/` 或 `common/`：公共结构和工具

2. 建立公共数据结构文件
   - 如果后续有链表、树相关题目，建议建立统一的：
     - `TreeNode`
     - `ListNode`
     - 辅助构造函数
   - 这样每个题目不用重复定义基础结构

3. 统一文件命名风格
   - 当前命名采用题号前缀，这很好
   - 建议继续保持：
     - `0167-twoSum.go`
     - `0231-isPowerOfTwo.go`
     - `0292-canWinNim.go`
   - 但建议在仓库层面固定一种命名规则，避免未来出现混搭

4. 控制历史版本残留
   - 现在看起来每道题通常只有一个实现，这很好
   - 如果以后要保留多个版本，建议放在明确的实验目录或用不同命名标识
   - 不要把多个近似实现长期混在根目录

## P2 提升代码质量

1. 优先使用更简洁的标准写法
   - `0292-canWinNim.go` 直接返回 `n % 4 != 0`，这是很好的写法
   - 这类题应尽量保持简洁，避免写成状态机式循环

2. 简化不必要的循环控制
   - `0231-isPowerOfTwo.go` 使用了 `for {}` + `break`，可读性一般
   - 这类题通常可以用更直接的条件表达式或位运算
   - Go 代码里建议优先追求“清晰 + 简短”

3. 减少重复计算
   - `0663-judgeSquareSum.go` 在循环里多次计算 `i*i + j*j`
   - 可以先存到临时变量，避免重复表达式，增强可读性

4. 保持函数接口一致
   - 当前函数返回值风格基本统一，都是直接返回结果
   - 这很好，建议继续保持
   - 对于有输入副作用的题目，测试里最好明确验证输入被修改的行为

5. 变量命名尽量表达语义
   - `left` / `right`、`i` / `j` 这类双指针变量没问题
   - 但像 `res`、`tmp` 这类变量在复杂题里最好尽量具体一些

## P3 提升测试质量

1. 使用表驱动测试
   - 目前测试已经采用了 Go 很推荐的表驱动风格
   - 这个方向是正确的，建议所有题都统一这种结构

2. 补充边界条件
   - `0167-twoSum_test.go` 的样例覆盖了部分情形，但还可以加：
     - 最小长度输入
     - 目标值在首尾
     - 重复元素
     - 无解场景

3. 为易出错题增加反例
   - `0231-isPowerOfTwo_test.go` 和 `0292-canWinNim_test.go` 都已经覆盖了一些边界值
   - 但仍建议补充更多负例，防止实现退化

4. 测试更明确地表达意图
   - 例如 `0663-judgeSquareSum_test.go` 可以补充更大的数和典型分解案例
   - 这样更容易验证双指针实现的正确性

5. 不要让测试隐式依赖实现细节
   - 如果某些函数会修改输入切片，测试应该显式断言这个行为
   - 如果不想依赖副作用，函数就应返回新切片或明确文档说明

## P4 建立更好的工程约定

1. 统一题解模板
   - 每新增一题都从同一个模板开始
   - 模板里建议包括：
     - 题号和题目名
     - 函数签名
     - 简短注释
     - 对应测试
     - 复杂度说明

2. 统一输入输出说明
   - 对于需要原地修改的题目，测试和注释里都要明确
   - 对于返回新结果的题目，避免使用可能引发副作用的写法

3. 补充公共帮助函数
   - 如果后续题目增多，可以加入：
     - 构造链表
     - 构造树
     - 比较切片
     - 辅助断言
   - 这样测试会更简洁

4. 控制注释噪音
   - 当前部分文件注释里有性能记录和题目说明
   - 性能记录可以保留在少数关键题里
   - 其余题解建议只保留必要的题目来源和核心思路

## P5 工具链优化

1. 强制使用 `gofmt`
   - 这是最重要的 Go 工具链优化
   - 可以作为提交前检查的一部分，避免手写排版差异

2. 增加静态检查
   - 可以考虑加 `golangci-lint` 或 `go vet`
   - 这对发现命名、重复逻辑和潜在 bug 很有帮助

3. 将测试执行标准化
   - 最好有一条固定命令可以跑完所有测试
   - 例如：`go test ./...`
   - 这样更容易在本地和 CI 保持一致

4. 文档收敛
   - 如果仓库里还会继续增长，建议增加一个简洁 `README.md`
   - 重点说明：
     - 如何新增题解
     - 如何命名文件
     - 如何运行测试
     - 如何格式化代码

## 具体值得先改的点

1. 统一所有文件的 `gofmt` 格式
2. 明确 `append(nums, nums...)` 的副作用语义
3. 给 `0231-isPowerOfTwo` 重写成更直接的实现
4. 给所有题目补充更完整的边界测试
5. 增加公共结构和测试辅助函数

## 推荐执行顺序

1. 先跑一遍 `gofmt` 统一格式
2. 再补测试边界和副作用验证
3. 然后整理目录和公共结构
4. 最后收敛工具链和文档

## 如果只做最有价值的 3 件事

1. 全仓库执行 `gofmt`
2. 补充边界测试和副作用测试
3. 建立公共数据结构和辅助函数
