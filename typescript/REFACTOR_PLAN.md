# TypeScript LeetCode 仓库重构方案

这个仓库的主体已经比较成熟，但目前存在一些典型的“长期刷题仓库”问题：配置混杂、入口分散、测试依赖构建产物、部分题解实现风格不统一。下面按优先级给出重构建议，目标是先提升稳定性，再提升可维护性，最后再优化体验。

## P0 先解决稳定性和配置冲突

1. 统一测试和编译链路
   - 当前 `package.json` 里同时有 `compile`、`pretest`、`posttest`、`test-ts`、`lint`、`fix` 等脚本
   - `pretest` 会先跑 `tsc`，`posttest` 会再跑 `gts lint`，这会让一次测试触发过多动作
   - 建议明确主流程，例如：
     - `test` 只负责单元测试
     - `lint` 只负责静态检查
     - `build` 只负责编译

2. 统一 ESLint / GTS / Prettier 的边界
   - 现在 `eslint.config.mjs`、`gts`、`.prettierrc.js` 同时存在
   - `eslint.config.mjs` 还在通过 `compat.extends("./node_modules/gts/")` 继承规则
   - 这会让规则来源不够直观，后续排查问题成本高
   - 建议确定谁是主配置源：
     - 如果以 `gts` 为主，就尽量减少额外手写规则
     - 如果以 `eslint` 为主，就把规则集中到 ESLint 配置里

3. 解决测试直接依赖 `build/` 产物的问题
   - `test/leetcode.test.js` 直接从 `../build/src/...` 导入
   - 这意味着测试必须先编译，路径也依赖输出目录结构
   - 这种方式不够稳，建议让 Jest 直接测 TypeScript 源码，或者至少统一一个稳定的构建测试入口

4. 检查 `test.json` 与 `tsconfig.json` 的职责分工
   - `tsconfig.json` 继承 `test.json`
   - 但当前还没看到清晰的“开发 / 测试 / 构建”三套配置边界
   - 建议拆成：
     - `tsconfig.json`：通用开发配置
     - `tsconfig.build.json`：只负责编译输出
     - `tsconfig.test.json`：只负责测试

## P1 规范目录结构

1. 明确 `src`、`test`、`build` 的角色
   - `src/`：题解源码
   - `test/`：单元测试
   - `build/`：编译输出
   - 这三个目录职责应该很清晰，避免测试直接依赖 `build` 作为源码入口

2. 统一题解文件命名
   - 目前文件名整体已经按题号排序，这一点很好
   - 建议继续保持统一风格：
     - `0001-two-sum.ts`
     - `0113-path-sum-ii.ts`
   - 尽量避免同一仓库内出现过多命名变体

3. 公共类型放到统一位置
   - `src/treeNode.ts` 已经是一个好的开始
   - 后续建议继续补充：
     - `ListNode`
     - 构造树/链表的辅助函数
     - 常用测试辅助函数
   - 这样每道题不会重复定义基础结构

4. 把实验性代码和正式题解分开
   - 如果某些文件只是对同一题的不同尝试，建议放到实验目录
   - 正式题解目录里尽量保持“一个题一个主版本”

## P2 提升代码一致性

1. 统一导出方式
   - 当前大多数题解是 `function foo(...) { ... } export { foo };`
   - 这个模式可以继续保留，但要确保所有文件都统一
   - 避免有的文件默认导出，有的文件命名导出，有的文件混用

2. 统一函数命名和实现风格
   - 例如 `twoSum`、`pathSum`、`check`、`intersection` 这类命名比较自然
   - 但同类题如果有多个版本，建议命名更明确：
     - `twoSumBruteForce`
     - `twoSumHashMap`
     - `pathSumDfs`

3. 收紧类型风格
   - 当前很多实现已经显式标注类型，这是优点
   - 但仍可以进一步统一：
     - 所有参数都标类型
     - 所有返回值都明确标注
     - 避免不必要的 `any`

4. 让注释更聚焦
   - 当前有题目链接、性能数据、中文说明等混合存在
   - 建议保留两类注释：
     - 题目来源
     - 对实现有帮助的解释
   - 其他历史说明可以迁移到文档中

## P3 提升测试质量

1. 把单测改成参数化风格
   - 当前 `test/leetcode.test.js` 是把多个题集中写在一个大文件里
   - 这种方式能跑，但维护性一般
   - 建议按题拆分，或者至少使用表驱动测试，减少重复样板

2. 让测试更接近真实使用
   - `test/0100-sameTree.test.js` 里同时使用字面量对象和 `TreeNode` 构造器
   - 这会让测试数据结构风格不一致
   - 建议统一通过公共 `TreeNode` 构造器创建测试树

3. 补边界条件
   - 现在很多测试已经覆盖常规样例，但还可以补：
     - 空数组
     - 单元素数组
     - 极端输入
     - 重复值
     - 无解情况

4. 减少重复测试代码
   - `test/leetcode.test.js` 中有重复测试块，例如 `0434-countSegments` 出现了两次
   - 这种重复容易让维护者误判覆盖情况
   - 建议通过参数化或拆分文件解决

5. 检查测试和源码的同步性
   - 由于测试依赖编译输出，源码改动后如果编译没跑，测试可能直接失效
   - 建议建立一个更直接的测试链路，减少这种耦合

## P4 提升公共能力

1. 建立标准题解模板
   - 每新增一题时都从同一个模板开始
   - 模板中建议固定包含：
     - 题号和题目名
     - 类型导入
     - `Solution` 形式或统一函数形式
     - 复杂度说明
     - 对应测试文件路径

2. 完善公共结构工具
   - `TreeNode` 已经有了
   - 后续可以补：
     - `buildTree`
     - `serializeTree`
     - `ListNode`
     - `buildList`
   - 这些工具能显著减少测试和调试成本

3. 保持“一个题一个主实现”
   - 如果保留多个版本，建议明确它们的差异和使用场景
   - 不要让仓库里长期存在没有说明的多个近似实现

4. 清理可以合并的简单题解
   - 对于逻辑特别简单的题，可以考虑合并风格或统一模块
   - 但不要为了“合并”牺牲题目和测试的一一对应关系

## P5 工具链收敛

1. 收敛到最小必要工具集合
   - 当前仓库中有 TypeScript、Jest、ESLint、GTS、Prettier、Babel
   - 对刷题仓库来说，工具已经偏多
   - 建议保留：
     - TypeScript：类型和编译
     - Jest：测试
     - ESLint + Prettier：静态检查和格式化
   - `gts` 可以保留，也可以逐步替换为更明确的自定义配置

2. 简化脚本入口
   - 建议最终只保留少量高频脚本：
     - `test`
     - `test:watch`
     - `lint`
     - `lint:fix`
     - `build`
     - `check`

3. 文档收敛
   - 可以把零散说明合并成一个简洁 `README.md`
   - 保留内容建议包括：
     - 如何添加新题
     - 如何运行测试
     - 如何编译
     - 目录约定

## 具体值得先改的点

1. `test/leetcode.test.js` 不要直接依赖 `build/src/...`
2. 拆分 `tsconfig`，明确开发/测试/构建职责
3. 统一 ESLint / GTS / Prettier 的配置边界
4. 把 `test/leetcode.test.js` 拆成按题号分文件或参数化测试
5. 补充公共数据结构辅助函数，减少测试重复代码

## 推荐执行顺序

1. 先修测试依赖和配置冲突
2. 再整理 `tsconfig`、Jest、ESLint 的职责边界
3. 然后拆分测试并补边界用例
4. 最后完善公共模块和文档

## 如果只做最有价值的 3 件事

1. 让 Jest 直接面向源码，不再依赖 `build/`
2. 统一 TypeScript 配置，拆出测试和构建的职责
3. 把大测试文件拆成更细的、可维护的测试结构
