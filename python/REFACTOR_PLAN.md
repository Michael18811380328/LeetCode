# Python LeetCode 仓库重构方案

这个方案按优先级排序，目标是先让仓库可维护，再让代码可扩展，最后再优化体验。

## P0 先修复基础问题

1. 修复 `utils.py` 里的明显错误
   - `TreeNode.__init__` 少了 `self`
   - 这是会直接影响本地运行的硬问题，优先级最高

2. 统一所有题解文件的可导入性
   - 现在很多文件名带编号和连字符，不适合 Python `import`
   - 建议至少保证测试环境能稳定导入，不依赖特殊路径技巧

3. 清理内建名覆盖
   - 避免使用 `dict`、`hash`、`list`、`str` 这类变量名
   - 这是低成本高收益的可读性优化

## P1 建立清晰目录结构

1. 拆分“题解代码”和“公共组件”
   - 目前 `utils.py` 把很多题解函数塞在一起，不利于维护
   - 建议拆成：
     - `solutions/`：每题一个文件
     - `common/`：`TreeNode`、`ListNode`、通用工具
     - `tests/`：单独测试目录

2. 统一文件命名风格
   - Python 层面建议全改成 `snake_case`
   - 例如：
     - `0100_is_same_tree.py`
     - `0237_delete_node.py`
     - `two_sum.py`
   - 这样导入、搜索、测试都更顺手

3. 删掉或隔离历史草稿代码
   - 现在看起来有些文件像“刷题过程中间产物”
   - 建议区分：
     - 最终版解法
     - 练习版/对比版
   - 不要把多个版本散落在 `src` 根目录

## P2 统一代码规范

1. 统一类型注解和导入
   - 所有使用 `List`、`Optional` 的文件都应显式导入
   - 这样本地、IDE、静态检查器才一致

2. 统一函数签名风格
   - LeetCode 风格可以保留，但仓库内部建议更一致
   - 比如：
     - `twoSum1(nums, target)` 改成 `two_sum_bruteforce(nums, target)`
     - `twoSum2(nums, target)` 改成 `two_sum_hashmap(nums, target)`

3. 统一注释语言和注释目的
   - 现在有中文、英文、题目链接、调试注释混杂
   - 建议只保留两类注释：
     - 题目来源/链接
     - 非显然的实现说明
   - 删除无效或者过时的调试备注

## P3 提升测试质量

1. 把测试改成参数化测试
   - 现在很多测试样例太少
   - 建议每题至少覆盖：
     - 标准输入
     - 边界输入
     - 最小规模输入
     - 特殊值输入

2. 补空值和异常情况测试
   - 例如树节点、链表节点、无解场景
   - 即使 LeetCode 不一定会传，也能验证本地实现更稳

3. 统一测试文件和实现文件的对应关系
   - 每个题解一个测试文件，或者按题号分组
   - 不要让测试依赖隐式路径或零散导入

4. 明确测试目标
   - 当前测试更像“跑通样例”
   - 建议升级成“验证题解正确性 + 防止回归”

## P4 提升公共能力

1. 建立通用数据结构模块
   - `TreeNode`
   - `ListNode`
   - `build_tree`
   - `build_list`
   - `tree_to_list`
   - 这些工具能大幅降低重复工作

2. 增加统一题解模板
   - 每个新题都从同一个模板开始
   - 模板里固定包含：
     - 问题链接
     - 复杂度说明
     - `Solution` 类
     - 必要的导入

3. 规范算法实现风格
   - 更偏向 Pythonic
   - 少写不必要的下标循环
   - 少写手动拼接和临时变量
   - 多用：
     - `enumerate`
     - `zip`
     - `any` / `all`
     - `set` / `dict`
     - 切片和列表表达式

## P5 工具链升级

1. 统一格式化工具
   - 如果只是想简单维护，`autopep8` 还能用
   - 但更推荐升级到：
     - `ruff` 做 lint
     - `black` 做格式化
     - `pytest` 做测试
   - 这样比现在手工记规则更可靠

2. 加入自动检查入口
   - 最好有一条命令能完成：
     - 格式检查
     - 静态检查
     - 单元测试
   - 例如一个 `make test` 或脚本入口

3. 文档收敛
   - 现在 `pylint.md`、`pytest.md` 偏说明文档
   - 可以合并成一个简洁的 `README.md`
   - 保留：
     - 如何运行测试
     - 如何添加新题
     - 代码规范

## 推荐执行顺序

1. 先修 `utils.py` 和导入问题
2. 再整理目录和命名
3. 然后统一类型注解和变量命名
4. 接着补测试
5. 最后再升级工具链

## 如果只做最值钱的 3 件事

1. 修复 `TreeNode.__init__` 和导入问题
2. 把测试改成参数化并补边界用例
3. 统一命名和删掉重复实现
